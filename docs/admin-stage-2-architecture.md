# VN-MASTERS: архитектура публичной демо-панели, этап 2

Дата реализации: 17 июля 2026 года.

## Назначение и границы

`admin.html` — публичная демонстрационная CRM-панель для показа будущих рабочих сценариев автосервиса. Это статическое приложение для GitHub Pages. Оно не является защищённой административной зоной и не должно восприниматься как готовая production-CRM.

Панель:

- не запрашивает логин или пароль;
- не подключает Firebase, Firestore, Firebase Authentication и другие облачные SDK;
- не отправляет данные на сервер;
- сохраняет рабочую базу только в IndexedDB текущего браузерного профиля;
- использует `localStorage` только для черновика заказ-наряда и не хранит там основную базу;
- позволяет перенести данные вручную через версионированную резервную копию JSON;
- содержит исключительно вымышленные демонстрационные данные с явным префиксом «Демо:».

Legacy-версия сохранена в `archive/admin-legacy.html`. Контрольный SHA-256:

`82817BD216C8B15A425AE6F5896DDEB1ABD16DA2088BA966D952FF1423C12581`.

## Устройство приложения

### Статический shell

`admin.html` содержит только семантический shell:

- боковую навигацию;
- верхнюю панель;
- предупреждение о локальном хранении;
- контейнер текущего раздела;
- общий диалог редактирования;
- onboarding;
- live-region уведомлений;
- скрытый input импорта JSON.

Контент разделов создаётся модульным JavaScript после открытия IndexedDB. Основная публичная часть сайта при этом не зависит от admin-модулей.

### CSS

- `assets/css/admin/admin.css` — дизайн-токены, базовый layout, sidebar, header и общая среда;
- `assets/css/admin/admin-components.css` — кнопки, формы, таблицы, статусы, метрики, диалоги, карточки и печать;
- `assets/css/admin/admin-responsive.css` — tablet/mobile layout от 1100 px до 320 px.

Admin-стили не подключаются к публичным страницам и не меняют существующую дизайн-систему сайта.

### JavaScript

Точка входа: `assets/js/admin/app.mjs`.

Ядро:

- `constants.mjs` — версии, статусы, store names и маршруты;
- `formatters.mjs` — деньги, количество, даты и телефоны;
- `utils.mjs` — UUID, нормализация, CSV, файлы, backup;
- `calculations.mjs` — чистые финансовые вычисления;
- `validation.mjs` — проверка сущностей и backup;
- `migrations.mjs` — обновление IndexedDB и JSON-схемы;
- `demo-data.mjs` — полностью вымышленный связанный набор данных;
- `state.mjs` — загруженное представление локальной базы;
- `router.mjs` — hash routing без персональных данных в URL;
- `ui.mjs` — безопасные DOM-компоненты через `textContent`.

Разделы:

- `modules/dashboard.mjs`;
- `modules/records.mjs`;
- `modules/work-orders.mjs`;
- `modules/finance.mjs`;
- `modules/directories.mjs`;
- `modules/settings.mjs`.

Хранилище:

- `storage/storage-adapter.mjs` — контракт;
- `storage/indexeddb-storage.mjs` — реализация IndexedDB;
- `storage/demo-storage.mjs` — фабрика локального адаптера.

Приложение не использует `innerHTML`, `eval` или `new Function` для пользовательских данных.

## Маршрутизация

Используются hash-маршруты:

- `#/dashboard`;
- `#/appointments`;
- `#/clients`;
- `#/vehicles`;
- `#/work-orders`;
- `#/payments`;
- `#/expenses`;
- `#/reports`;
- `#/directories`;
- `#/employees`;
- `#/trash`;
- `#/settings`.

ID, телефоны и другие данные не помещаются в URL. Редактирование происходит в диалоге поверх раздела.

## Жизненный цикл

1. Открывается `admin.html`.
2. IndexedDB обновляется до `DB_VERSION`.
3. При первом запуске пользователь выбирает пример базы или пустую базу.
4. Все store загружаются в `AdminState`.
5. Раздел строится из DOM-узлов.
6. Создание и изменение проходят через валидацию и storage adapter.
7. После операции обновляются только нужные store и журнал.
8. Удаление основных записей сначала перемещает их в корзину.

## Целостность операций

- у каждой записи есть самостоятельный UUID или стабильный импортированный ID;
- телефон, VIN, госномер и номер заказа не используются как первичный ключ;
- редактирование заказа обновляет существующий заказ и его строки в одной IndexedDB-транзакции;
- удалённые из редактора строки заказа удаляются из связанного набора, поэтому повторное сохранение не создаёт дубли;
- визиты, начисления и долг клиента вычисляются из заказов и платежей, а не накапливаются счётчиком;
- удаление клиента блокируется при связанных автомобилях или заказах;
- удаление автомобиля блокируется при связанных заказах;
- безвозвратное удаление заказа очищает его строки и платежи.

## Черновик заказ-наряда

Черновик хранится в `localStorage` по ключу версии `vnmasters.admin.workOrderDraft.v2`.

Он сохраняется:

- с debounce во время ввода;
- по `Ctrl+S` / `Cmd+S`;
- при скрытии вкладки;
- при закрытии диалога.

При следующем создании заказа панель предлагает восстановление. После успешного сохранения черновик удаляется.

## Import / export

JSON содержит:

- идентификатор продукта;
- `schemaVersion`;
- время экспорта;
- флаг `demoOnly`;
- все store;
- сводное количество записей;
- настройки.

Перед импортом файл проверяется, при необходимости мигрируется и только затем одной транзакцией заменяет базу. До замены панель автоматически скачивает текущую резервную копию.

CSV доступен для клиентов, автомобилей, записей, заказов, платежей, расходов и отчёта. CSV предназначен для просмотра и ручного анализа, а не для обратного импорта.

## Подготовка к будущему Firestore

UI и доменная логика не обращаются к Firebase напрямую. Доступ к данным изолирован контрактом `StorageAdapter`, поэтому на следующем отдельном этапе можно реализовать `FirestoreStorageAdapter`, сохранив сущности, стабильные ID, расчётные функции и интерфейс.

При таком переходе потребуются Google Sign-In, Firebase UID, серверно проверяемые роли и Firestore Security Rules. Клиентская проверка роли не считается защитой. Миграция локальной демо-базы должна выполняться отдельным контролируемым инструментом после входа и не должна автоматически публиковать данные из браузера.

## Проверки

Автоматически проверяются:

- финансовые формулы;
- частичная оплата, переплата и возврат;
- отсутствие накопления суммы при редактировании;
- отсутствие двойного вычитания связанных закупок;
- нормализация денег, телефона, VIN и госномера;
- миграции backup;
- связи демонстрационной базы;
- наличие локальных модулей и ресурсов;
- отсутствие Firebase, password input, внешних SDK и небезопасной HTML-вставки;
- сохранность legacy-файла.

Браузерные проверки выполнены через Playwright на 1440×900, 1280×800, 1024×768, 390×844, 375×812, 360×800 и 320×568.
