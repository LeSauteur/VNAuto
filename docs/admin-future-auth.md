# VN-MASTERS: требования к будущей защищённой CRM

Публичная демо-панель намеренно работает без авторизации и сервера. Добавление формы логина поверх IndexedDB не сделает её защищённой системой: код, база и проверки остаются под контролем пользователя браузера.

## Что нельзя переносить в production без переработки

- IndexedDB как единственный источник истины;
- локальный audit log;
- клиентские проверки прав;
- отсутствие серверной резервной копии;
- свободный JSON-импорт без административного workflow;
- общие демо-справочники без организации/филиала;
- публичный URL `admin.html`;
- хранение реальных персональных данных в публичной демонстрации.

## Целевая схема следующего оплаченного этапа

Для запланированной экосистемы VN-MASTERS целевым вариантом может быть Firebase:

- Google Sign-In через Firebase Authentication;
- Firebase UID как идентификатор пользователя, но не как идентификатор сотрудника или клиента;
- Firestore как серверный источник истины;
- Cloud Functions или другой доверенный серверный слой для денежных и административных операций, где одной Security Rules недостаточно;
- Firebase App Check как дополнительный слой, не заменяющий Authentication и Rules.

Ни один из этих компонентов в публичную демо-панель не подключён.

### Firestore и серверные операции

- Firestore-коллекции с документированными связями и обязательным `organizationId`;
- Firestore-транзакции для связанных изменений;
- Cloud Functions/доверенный API для возвратов, назначения ролей, импорта и других важных действий;
- версионированные миграции;
- фоновые резервные копии с проверкой восстановления;
- отдельные окружения development, staging и production;
- журнал безопасности, недоступный для изменения обычному пользователю.

### Google Sign-In и идентификация

- Google Sign-In через Firebase Authentication;
- Firebase UID в профиле членства организации;
- MFA для собственника и администраторов;
- ограничение попыток входа;
- короткоживущие сессии и безопасная ротация;
- отзыв сессий;
- безопасное восстановление доступа;
- журнал входов и критических действий.

Нельзя хранить пароли, хеши или секреты в HTML/JavaScript/localStorage.

### Роли и Firestore Security Rules

Минимальные роли:

- `owner`;
- `manager`;
- `mechanic`;
- `accountant`.

Пользователь не может самостоятельно назначить или повысить себе роль. Первичный `owner` создаётся только контролируемой административной процедурой. Последующие роли назначает `owner` через доверенную серверную операцию.

Firestore Security Rules должны проверять как минимум:

- наличие аутентификации;
- членство UID в нужной организации;
- роль членства;
- совпадение `organizationId` документа;
- допустимые поля и переходы состояния;
- запрет клиентской записи в защищённый audit log;
- запрет прямого изменения role/owner через обычный клиент.

Права должны проверяться Rules и, для критических операций, доверенным серверным кодом. Следует отдельно определить доступ к:

- персональным данным клиентов;
- финансовым отчётам;
- изменению и отмене платежей;
- возвратам;
- закупочным ценам;
- сотрудникам;
- экспорту;
- безвозвратному удалению;
- настройкам организации.

### Организационная модель

Даже для одного сервиса полезно заложить:

- `organizationId`;
- `locationId`;
- часовой пояс;
- нумерацию документов;
- юридические реквизиты;
- настройки валюты и налогов;
- пользователей и членство в организации.

### Адаптер Firestore

Текущий `IndexedDbStorage` должен быть заменён реализацией `FirestoreStorageAdapter` того же доменного контракта. UI не должен импортировать Firebase SDK напрямую в каждый модуль.

Адаптер отвечает за:

- чтение и подписки на коллекции;
- транзакционное сохранение заказа и строк;
- optimistic locking/version conflicts;
- серверные timestamps;
- пагинацию;
- обработку offline-состояния без превращения клиентского кэша в источник истины;
- вызов доверенных операций для возвратов, ролей, импорта и удаления.

### Данные

Серверная модель должна сохранить принципы демо:

- стабильные ID;
- отдельные сущности клиентов, автомобилей, заказов, строк, платежей и расходов;
- деньги в минимальных единицах;
- идемпотентные операции;
- вычисляемый статус оплаты;
- возврат отдельной операцией;
- транзакционное сохранение заказа;
- мягкое удаление с политикой retention;
- версии записей для optimistic locking.

Для денежных операций рекомендуется append-only ledger: исправление проводится обратной/корректирующей операцией, а не незаметной заменой истории.

## Безопасность персональных данных

До ввода реальных данных необходимо определить:

- правовое основание и тексты согласия;
- срок хранения;
- порядок исправления и удаления;
- перечень сотрудников с доступом;
- политику экспорта;
- шифрование транспорта и серверного хранилища;
- управление секретами;
- мониторинг утечек;
- журналирование просмотров и выгрузок;
- обработку инцидентов.

## Миграция локальной демонстрационной базы

Импорт JSON в Firestore нельзя выполнять автоматически при входе. Нужен отдельный workflow:

1. пользователь входит через Google Sign-In;
2. подтверждается его организация и роль `owner`;
3. JSON проверяется по версии и показывается предварительный отчёт;
4. пользователь подтверждает, что файл не содержит чужих данных;
5. импорт выполняется доверенным серверным кодом с idempotency key;
6. связи и суммы сверяются;
7. создаётся защищённая запись журнала;
8. исходный файл не хранится дольше необходимого.

## Переход от демо к production

1. Зафиксировать подтверждённые бизнес-процессы и роли.
2. Утвердить серверную модель и финансовые правила.
3. Создать Firebase-проект, Firestore-модель, Cloud Functions и миграции.
4. Перенести чистые расчётные функции в общий протестированный доменный слой.
5. Заменить IndexedDB adapter на Firestore adapter.
6. Добавить Google Sign-In, Firebase UID, роли и Security Rules.
7. Реализовать импорт демо-JSON только как контролируемый административный инструмент.
8. Провести security review, нагрузочные и recovery-тесты.
9. Только после этого разрешать реальные персональные и финансовые данные.

## Что можно повторно использовать

- информационную архитектуру разделов;
- визуальную систему и адаптивные компоненты;
- модель заказ-наряда и отдельных строк;
- финансовые формулы после бизнес-подтверждения;
- валидацию как клиентскую подсказку;
- формат резервной копии как источник для миграционного конвертера;
- тестовые сценарии CRUD и финансов.

Клиентская валидация при этом остаётся только UX-слоем; сервер обязан повторять все критические проверки.
